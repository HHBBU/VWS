```python
"""
Historical data module — uses the static 30-year dataset from the Student Guide.
Students see this data in the simulation and forecast Year 31.
"""

def generate_historical_demand(seed=None):
    """
    Return 30 years of historical demand for SKU A and SKU B.
    This is the same table printed in the Student Guidebook.
    Seed parameter is accepted but ignored (data is static).
    """
    years = list(range(1, 31))
    
    sku_a = [
        8100, 8450, 8700, 9050, 9200, 9600, 9750, 10100, 10350, 10650,
        10900, 11200, 11450, 11750, 12050, 12350, 12650, 12900, 13200, 13500,
        13850, 14150, 14450, 14800, 15050, 15350, 15650, 16050, 16350, 16700
    ]
    
    sku_b = [
        9150, 8750, 8900, 9200, 9000, 8750, 9250, 9050, 8850, 9300,
        9000, 8800, 9150, 8950, 9200, 9050, 8900, 9250, 8850, 9150,
        8950, 9100, 8800, 9200, 9000, 8850, 8800, 9000, 9200, 9000
    ]
    
    # Calculate trend for display
    # SKU A: strong upward trend ~280 units/year
    # SKU B: stable around 9,000
    n = len(years)
    sum_x = sum(years)
    sum_y_a = sum(sku_a)
    sum_xy_a = sum(x * y for x, y in zip(years, sku_a))
    sum_x2 = sum(x * x for x in years)
    trend_a = round((n * sum_xy_a - sum_x * sum_y_a) / (n * sum_x2 - sum_x * sum_x))
    
    sum_y_b = sum(sku_b)
    sum_xy_b = sum(x * y for x, y in zip(years, sku_b))
    trend_b = round((n * sum_xy_b - sum_x * sum_y_b) / (n * sum_x2 - sum_x * sum_x))
    
    return {
        'months': years,  # kept as 'months' key for backward compatibility
        'sku_a': sku_a,
        'sku_b': sku_b,
        'trend_a': trend_a,
        'trend_b': trend_b,
        'data_points': list(zip(years, sku_a, sku_b))
    }


def get_forecast_data_for_display():
    """
    Returns historical data formatted for display in templates.
    """
    import numpy as np
    data = generate_historical_demand()
    
    chart_data = {
        'labels': [f'Year {y}' for y in data['months']],
        'datasets': [
            {
                'label': 'SKU A (Trend Tee)',
                'data': data['sku_a'],
                'borderColor': '#667eea',
                'fill': False
            },
            {
                'label': 'SKU B (Core Jogger)',
                'data': data['sku_b'],
                'borderColor': '#10b981',
                'fill': False
            }
        ]
    }
    
    return {
        'chart_data': chart_data,
        'raw_data': data,
        'summary': {
            'avg_a': int(np.mean(data['sku_a'])),
            'avg_b': int(np.mean(data['sku_b'])),
            'trend_a': data['trend_a'],
            'trend_b': data['trend_b']
        }
    }
```

## FIX 2 (LOW — OPTIONAL): Update M2 demand derivation if using Option B

If you apply Option B above, the M1 forecasts are now ANNUAL (e.g., ~17,800 units/year).
Module 2 currently converts to daily demand with:
```python
daily_demand_baseline_A = m1_forecast_A / 30.0  # assumes MONTHLY forecast
```

For annual data, this should be:
```python
daily_demand_baseline_A = m1_forecast_A / 365.0  # ANNUAL forecast → daily
```

**BUT** — this would make daily demand ~49 units/day instead of ~593, which is 12× lower. This breaks M2's capacity calibration (Standard = 800/day was designed for ~900/day demand).

**Two sub-options:**

**Sub-option B1:** Keep the `/30.0` divisor. This means M1 forecasts are treated as "monthly equivalent" internally even though the guide calls them "Year 31." The math works because the guide's Year 31 values (~17,000) are in the same magnitude as the original Month 25 values (~18,000). Slightly misleading but functionally correct.

**Sub-option B2:** Change to `/365.0` AND rescale M2 capacity down by 12× to match. This is a bigger change that cascades through M2 and M3. **Not recommended** unless you want to rebuild M2.

**My recommendation:** Use **Option B + Sub-option B1**. The forecast numbers happen to be in the same ballpark, so the simulation works correctly. Students won't notice because they don't see the internal `/30.0` calculation.
